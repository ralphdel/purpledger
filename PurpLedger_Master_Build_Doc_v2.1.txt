1. PRODUCT OVERVIEW & CORE PHILOSOPHY
	

PurpLedger is a dual-mode invoicing and payment collection platform. It is the only invoicing tool in its class that serves two fundamentally different workflows under one roof: structured bookkeeping for merchants who collect offline, and live digital payment collection for merchants who collect via Paystack. Both modes are first-class citizens of the platform.


1.1 The Two Invoice Types — The Core Differentiator


	Record Invoice   vs   Collection Invoice
	Purpose
	Bookkeeping and record-keeping only. For merchants who receive offline cash, bank transfers, or any payment outside Paystack. The invoice documents what was agreed and tracks what has been received.
	Payment Link
	None. No Paystack integration. No QR code for payment.
	Amount Paid
	Manually entered by the merchant when payment is received. Supports partial entries — merchant updates it progressively as client pays.
	Reminders
	Automated reminder emails sent to client using a 'contact merchant to pay' template — no Pay Now button.
	Status logic
	Open → Partially Paid → Closed (merchant manually marks as closed or enters full amount). Manual Close protocol (reason required) applies.
	Available to
	All paid plans (Individual + Corporate) + Starter (up to 5 records).
	



	Collection Invoice
	Purpose
	Live digital payment collection. The merchant sends a link; the client pays online via Paystack (card, bank transfer, USSD).
	Payment Link
	Auto-generated short link + QR code. Customer opens link, edits amount if partial, pays on Paystack.
	Amount Paid
	Auto-updated via Paystack webhook on every successful transaction.
	Reminders
	Automated reminder emails with Pay Now button linking to the payment portal.
	Status logic
	Full proportional recalculation engine. Open → Partially Paid → Closed (auto on full payment). Manual Close protocol applies.
	Available to
	Individual plan (BVN verified) + Corporate plan (all docs verified). Starter cannot use Collection Invoices.

3. ONBOARDING FLOW
Landing Page → Payment → Account Activation → Dashboard
	

The onboarding flow is designed to convert landing page visitors into active merchants with minimal friction. Payment is collected before account creation, ensuring only committed merchants enter the platform. Supabase magic links are used throughout — no passwords are ever transmitted in emails.


3.1 Starter Plan Onboarding
STARTER ONBOARDING — Free, No Payment
Landing page → 'Start Free' CTA → /register (standard signup form)


Step 1: Signup Form
  Fields: Business Name (required) + Email (required) + Password (required)
  Supabase createUser() fires on submit
  merchants row created: subscription_plan='starter', verification_status='unverified'


Step 2: Email Verification
  Supabase sends standard verification email (magic link)
  Page shown: 'Check your inbox to verify your email address.'


Step 3: Verification Click
  Merchant clicks link in email → Supabase confirms email → redirected to /dashboard
  Dashboard opens in Starter-gated experience


Total steps: 3. No payment, no KYC, immediate access.
	

3.2 Individual & Corporate Plan Onboarding
Subscription Fee — PurpLedger Absorbs Paystack Fee
When a merchant pays their subscription (₦5,000 or ₦20,000), the Paystack processing fee (~₦175 on ₦5,000) is absorbed by PurpLedger. The merchant is charged the clean published price. This is baked into pricing and is industry-standard SaaS practice. The subscription payment goes to PurpLedger's main Paystack account — not to any merchant subaccount.
	

Brevo SMTP — Already Configured
Supabase custom SMTP has been configured to route all auth emails (magic links, password resets, email verification) through Brevo's SMTP servers. This removes Supabase's 3 emails/hour free-tier limit. Brevo free tier: 300 emails/day. No code change required — this is a Supabase Dashboard configuration already applied. All onboarding magic links are delivered via Brevo.
	

PAID PLAN ONBOARDING — Individual or Corporate
Landing page → Select plan (Individual or Corporate) → /onboarding/[plan]


STEP 1 — Intent Capture (/onboarding/[plan])
  Fields: Business Name + Email
  No password collected here
  System checks: does email already exist in auth.users?
    YES → 'This email already has an account. Log in to upgrade your plan.'
    NO  → Continue
  Submit creates a PENDING_PAYMENT record in onboarding_sessions table:
    { email, business_name, plan, status: 'awaiting_payment', expires_at: now()+24h }


STEP 2 — Payment Initialization
  System calls PaymentService.initializeTransaction() for the subscription fee
  This charges PurpLedger's OWN Paystack main account (not a subaccount)
  Merchant is redirected to Paystack checkout for the subscription fee


STEP 3 — Payment Webhook (charge.success)
  Paystack fires charge.success to /api/webhooks/paystack
  Handler detects this is a subscription payment (metadata.type = 'subscription')
  Creates Supabase auth user (email only, NO password set yet)
  Creates merchants row: subscription_plan=[plan], verification_status='unverified'
  Updates onboarding_sessions: status='payment_confirmed'
  Triggers Brevo: Welcome + Set Password email


STEP 4 — Welcome Email (Brevo Template: onboarding_welcome)
  Subject: 'Welcome to PurpLedger — activate your account'
  Body: Business name greeting + plan confirmation
  CTA: 'Set Your Password' → links to Supabase magic link (secure, time-limited)
  NO password in the email. The magic link IS the credential.


STEP 5 — Set Password Page (/onboarding/set-password)
  Merchant arrives via magic link (Supabase handles token validation)
  Single form: New Password + Confirm Password (min 8 chars, strength indicator)
  On save: Supabase updateUser({ password }) called
  Merchant is now fully authenticated


STEP 6 — Dashboard (Plan-Gated)
  Redirected to /dashboard
  Dashboard loads with correct plan permissions
  Persistent KYC banner if on Individual/Corporate (verification required for payment links)
  Onboarding checklist shown (collapsible): Complete Profile / Verify Identity / Add Settlement Account
	

3.3 Payment Failure Handling
Failure Scenario
	System Behaviour
	Customer exits Paystack without paying
	onboarding_sessions record expires after 24h. No auth user created. Merchant can restart from landing page.
	Paystack payment fails (card declined)
	Paystack redirect to callback URL with status=failed. Page shown: 'Payment unsuccessful. Please try again.' Retry button re-initializes a new Paystack transaction.
	Webhook not received (Paystack delay)
	onboarding_sessions.status remains 'awaiting_payment'. Paystack retries webhook for up to 72h. A cron job checks pending sessions and reconciles via PaymentService.verifyTransaction() after 30 min.
	Magic link expires before merchant clicks
	Merchant can request a new one from the 'Resend activation email' page at /onboarding/resend. Auth user already exists — system sends a new Supabase magic link.
	Duplicate payment (merchant pays twice)
	Idempotency key on onboarding_sessions prevents duplicate account creation. Second charge.success is caught and refunded via Paystack refund API.
	

3.4 Onboarding Sessions Table (New)
CREATE TABLE IF NOT EXISTS onboarding_sessions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email           TEXT NOT NULL,
  business_name   TEXT NOT NULL,
  plan            TEXT NOT NULL,   -- 'individual' | 'corporate'
  status          TEXT NOT NULL DEFAULT 'awaiting_payment',
  --  'awaiting_payment' | 'payment_confirmed' | 'activated' | 'expired'
  paystack_ref    TEXT,            -- Paystack transaction reference
  amount_paid     NUMERIC(10,2),   -- Subscription fee paid
  merchant_id     UUID REFERENCES merchants(id),  -- Set after account creation
  expires_at      TIMESTAMPTZ NOT NULL,   -- now() + INTERVAL '24 hours'
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  idempotency_key TEXT UNIQUE,     -- Paystack ref — prevents duplicate processing


  CONSTRAINT valid_plan CHECK (plan IN ('individual', 'corporate')),
  CONSTRAINT valid_status CHECK (status IN (
    'awaiting_payment','payment_confirmed','activated','expired'
  ))
);


-- Cron reconciliation index
CREATE INDEX idx_onboarding_pending
  ON onboarding_sessions(status, created_at)
  WHERE status = 'awaiting_payment';

4.3 Record Invoice — Amount Paid Update Flow
A verified merchant receives a cash or bank transfer from a client. They update PurpLedger to keep their records accurate.
1. Merchant opens the Record Invoice from their invoice list.
2. Invoice detail page shows: Grand Total | Amount Paid (current) | Outstanding Balance.
3. Merchant clicks 'Record Payment' button.
4. A drawer/modal opens: 'Record a Payment Received'
   * Amount Received (required — pre-fills with outstanding balance)
   * Payment Method: Cash / Bank Transfer / Cheque / Other (dropdown)
   * Date Received (date picker, defaults to today)
   * Reference / Note (optional freetext)
5. Merchant clicks 'Save Payment'.
6. Server Action creates a manual_payments record and updates invoice.amount_paid.
7. Invoice status transitions: Open → Partially Paid → Closed (when amount_paid = grand_total).
8. If partially paid: Outstanding Balance updates. Reminder schedule continues.
9. If fully paid: Invoice closes. Reminders stop. Merchant sees 'Paid in Full' badge.


4.4 Record Invoice Status State Machine
Status
	Trigger
	OPEN
	Invoice created. amount_paid = 0. Due date has not passed.
	PARTIALLY PAID
	Merchant records a payment. amount_paid > 0 but < grand_total.
	CLOSED
	Merchant records payment that brings amount_paid = grand_total (system auto-closes).
	MANUALLY CLOSED
	Merchant clicks 'Close Manually'. Requires mandatory Reason Code. Balance may be non-zero (e.g., small amount waived).
	OVERDUE
	Due date has passed. amount_paid < grand_total. Reminders intensify (frequency increases). Link to 'Mark as Bad Debt' (manual close with reason: Bad Debt Write-Off).
	VOID
	Merchant voids invoice pre-payment. Mandatory reason required. amount_paid must be 0.

CREATE TABLE IF NOT EXISTS manual_payments (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  invoice_id      UUID NOT NULL REFERENCES invoices(id) ON DELETE RESTRICT,
  merchant_id     UUID NOT NULL REFERENCES merchants(id),
  amount          NUMERIC(12,2) NOT NULL CHECK (amount > 0),

  reference_note  TEXT,          -- Merchant's internal note
  recorded_by     UUID REFERENCES auth.users(id),  -- Which team member recorded it
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);


-- RLS
ALTER TABLE manual_payments ENABLE ROW LEVEL SECURITY;
CREATE POLICY manual_payments_merchant ON manual_payments
  FOR ALL USING (
    merchant_id IN (
      SELECT id FROM merchants WHERE user_id = auth.uid()
      UNION
      SELECT merchant_id FROM merchant_team
        WHERE user_id = auth.uid() AND is_active = true
    )
  );

7. SETTLEMENT ARCHITECTURE & PAYMENTSERVICE
Processor-Agnostic Design — Do Not Call Paystack Directly

7.1 The Abstraction Rule
Non-Negotiable Architecture Constraint
src/lib/payment/ is the only place in the entire codebase that knows Paystack exists. No route handler, Server Action, or component imports the Paystack SDK or calls api.paystack.co directly. Switching to Monnify in v1.3 means writing one new adapter file — zero changes to invoice logic, webhook handling, or Server Actions.
Test: grep the codebase for 'paystack.co' or 'PAYSTACK_SECRET_KEY'. Only PaystackAdapter.ts and .env.local should match. Any other match is a violation.

14. WHATSAPP REMINDERS

WhatsApp is the dominant communication channel in Nigeria. User feedback strongly indicates that WhatsApp reminders will have significantly higher open and response rates than email reminders. This section specifies the WhatsApp reminder system using WATI (WhatsApp Business API wrapper) as the MVP provider.


14.1 Provider Decision — WATI
Factor
	Decision
	Provider
	WATI (wati.io) — WhatsApp Business API wrapper popular in Nigeria. Supports template messages, contact management, and REST API.
	Why WATI over direct Meta API
	Faster setup (days vs weeks), pre-built template management UI, Nigerian market support, reasonable per-message pricing. Direct Meta API is the long-term goal — WATI is the bridge.
	Message type
	Template messages only (WhatsApp Business API requirement). All reminder templates must be pre-approved by Meta via WATI's template submission flow. Plan 5–10 business days for approval.
	Cost
	WATI subscription + per-conversation fee (~$0.02–0.05 per 24h conversation window). Volume-based.
	Environment variable
	WATI_API_URL, WATI_API_TOKEN — added to .env.local and Vercel environment.

  // Email channel

    send Brevo template (COLLECTION_REMINDER or RECORD_REMINDER)


  // WhatsApp channel
  if client.whatsapp_number is set AND merchant.subscription_plan != 'starter':
    if invoice.invoice_type == 'collection':
      send WATI template COLLECTION_REMINDER_WA
    else:
      send WATI template RECORD_REMINDER_WA


  // If neither channel available:
  log warning: 'No contact channel for client [id] on invoice [id]'
  // Surface this in merchant dashboard: 'Add email or WhatsApp to send reminders'


  // Both channels can fire simultaneously — not either/or
