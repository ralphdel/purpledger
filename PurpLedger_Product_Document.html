1. Product Overview

PurpLedger is a fintech SaaS platform built for Nigerian SMEs and freelancers to create invoices, track partial payments, manage team access, and receive payments online. The core differentiator is its proportional payment engine — when a client makes a partial payment, every line item (subtotal, tax, discount, and Paystack fee) is settled proportionally, ensuring financial accuracy at every step.

Mission

To eliminate spreadsheet-based invoice tracking and give every small business a professional, compliance-first invoicing and payment system with no technical overhead.

Key Architectural Decisions

  • Server Actions over API Routes for mutations — All form submissions and data mutations use "use server" Server Actions in src/lib/actions.ts. This eliminates CSRF concerns and keeps logic server-side.
  • Service Client for admin operations — getServiceClient() (Supabase service role key) bypasses RLS only inside trusted server actions where application-level scoping (.eq("merchant_id", ...)) provides the security boundary.
  • Standard client for reads — createClient() uses the session-scoped Supabase client for reads; RLS policies ensure data isolation between merchants.
  • Proportional Payment is server-enforced — calculateProportionalPayment() in calculations.ts runs on the server inside the webhook handler and is never trusted from the client.

    Row Level Security (RLS) Strategy

    • Merchants can only SELECT/UPDATE their own row.
    • Invoices/Clients/Transactions are scoped to merchant_id = auth.uid() via RLS.
    • Admin operations (suspend, delete, verify) use the Supabase service role key inside Next.js Server Actions — never exposed client-side.
    • Team members see data under their employer's merchant_id, not their own.

    7. Security Architecture

    Authentication Flow

  • Sessions managed by Supabase JWTs with httpOnly cookies
  • Middleware (src/middleware.ts) validates every protected route:

    • - Unauthenticated → /login

    - must_change_password = true → /set-password

    - Admin routes check actor_role = "admin" in the merchants table

    - Suspended merchants → /suspended

    Access Control

  • Merchants can only access their own data (enforced at DB + app level)
  • Team Members are scoped to their employer's merchant_id
  • Super Admins use Supabase service role only inside server-side actions
  • Public Payment Portal (/pay/*) requires no authentication — data is validated server-side by invoice ID

    • One-Time Temp Password Flow

  • On invite, a cryptographically random temp password is generated
  • Supabase auth user created with email_confirm: true (bypasses email OTP)
  • must_change_password = true stored in merchant_team
  • On first login, middleware intercepts and redirects to /set-password
  • After setting new password, must_change_password cleared in DB

    • Data Integrity

  • Invoice outstanding_balance is computed server-side from transactions
  • k_factor (proportional ratio) is stored per-transaction for auditability
  • Paystack paystack_reference field prevents duplicate payment recording
  • Manual close reasons are from a locked enum to prevent arbitrary entries
